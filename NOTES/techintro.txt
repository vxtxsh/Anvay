4 routes defined in app.routes.ts: /dashboard, /event-management, /club-management, /students (default redirects to /dashboard)

1. What is a Bearer Token?
A Bearer Token is a type of security token that gives access to the "bearer" (the person or application holding it).
Imagine a "Bearer Bond" or a movie ticket: the theater doesn't care who bought the ticket; they only care that you are currently holding (bearing) a valid ticket. If you have it, you get in.
How it works in Web Apps:
The Format: It is usually a JWT (JSON Web Token)—a long string of encrypted characters.
The Header: When your Angular app talks to your backend, it sends this token in the HTTP Header like this: Authorization: Bearer <your_very_long_token_string>
The Risk: Because anyone who holds the token can access the data, it must be sent over HTTPS to prevent "man-in-the-middle" attacks.
2. What is an Interceptor?
In Angular, an Interceptor is a piece of code that sits between your application and the internet. It "intercepts" outgoing requests or incoming responses before they reach their final destination.
Why do we use it here?
Without an interceptor, if you had 50 different API calls (getting user profile, fetching posts, updating settings), you would have to manually add the Authorization header to every single one. That is repetitive and prone to bugs.
The Interceptor acts like a "Standardized Shipping Label" machine:
Intercept: You click "Save Profile."
Process: The Interceptor pauses the request for a split second.
Modify: It looks in your local storage, finds your Bearer Token, and "clones" the request with the Authorization header attached.
Send: It releases the request to the server.
The Code Logic You Shared:
TypeScript
const jwtInterceptor: HttpInterceptorFn = (req, next) => {
 const token = authService.getToken();
 if (token) {
   // We CLONE because the original request is "Read-Only"
   req = req.clone({ setHeaders: { Authorization: `Bearer ${token}` } });
 }
 return next(req); // Send the modified request on its way
};

App component
 
Now that you've looked at the main.ts (the ignition) and app.config.ts (the settings), the AppComponent is the Main Stage.
Anatomy of the Root Component
1. The Selectors and Metadata
selector: 'app-root': This matches the <app-root></app-root> tag in your index.html. Angular looks for that tag and injects this component’s template right there.
standalone: true: This confirms the component doesn't need a middleman (NgModule). It manages its own dependencies.
imports: [...]: Since this is a standalone component, you must explicitly list the tools it uses. You’ve imported RouterOutlet (for navigation) and ToastComponent (for notifications).
2. The Template: Two Key Elements
Your template is short but incredibly powerful because of what these two tags represent:
<app-toast></app-toast>: By placing this here, your toast/notification system is global. It sits at the root level, meaning it can stay visible even when you navigate between different pages. It’s "above" the rest of the app.
<router-outlet></router-outlet>: Think of this as a dynamic placeholder. When the user goes to /login, the Login component appears here. When they go to /dashboard, the Dashboard component replaces it. The AppComponent itself usually stays static (headers, footers, toasts), while the router-outlet swaps the content.
7. app.routes.ts
Purpose: Defines all the routes (pages) in the app — what component loads when you visit each URL.
Why it's needed:
Maps URLs like /students, /events, /dashboard to their components

If yes → loads StudentsComponent
If no → redirects to /login
